一、项目简介、项目目录和文件结构说明
PetLife 宠物生活馆项目围绕“商品购买 + 服务预约 + 宠物档案 + 会员权益 + 后台运营”展开，面向真实养宠场景设计移动端 Web 应用。当前项目包含用户前台、后台管理端和 Node/SQLite 后端的完整示例项目，具备前后端联调、数据持久化和基础内容运营能力。
从应用意义上看，该项目不再停留在普通商城页面展示，而是把商品零售、洗护美容等服务、宠物档案管理和后台内容运维整合在同一体系中，能够较完整地体现移动项目开发中常见的业务分析、页面组织、状态管理、接口设计、数据库设计和测试验证全过程。

二、需求分析
本项目的用户可以分为普通用户和管理员两类。普通用户主要关注选购商品、预约服务、维护宠物档案和查看会员权益；管理员主要关注商品目录维护、门店与时段配置以及订单履约。由于系统同时支持商品链路和服务链路，所以需求分析必须强调不同场景下的功能差异。

三、项目分析与设计
项目整体采用三端协同架构：移动端前台负责用户体验，后台管理端负责目录维护和运营数据处理，Node.js 后端负责 API、业务规则和 SQLite 持久化。这样既能保持移动端页面开发效率，也能保证订单、预约和库存等关键数据有统一的后端来源。

图 1  项目总体架构图
在页面设计上，移动端将首页、分类、商品、服务、订单、我的等页面统一纳入 AppShell，底部 TabBar 提供主导航，详情页和确认页则通过独立路由承载深层业务操作。

图 2  移动端页面结构图
首页是项目的核心入口页，既承担品牌展示，也承担业务分流。根据 `HomeView.vue` 的真实实现，首页模块结构由主视觉、宠物切换、快捷入口、热门服务、组合推荐、热卖商品和会员信息组成。

图 3  首页功能模块结构图
服务端采用典型的分层设计，路由层只负责 URL 入口，控制器层负责请求/响应，服务层负责业务规则，repository 层负责 SQL 访问。这种结构能减少耦合，便于后续继续扩展后台运营能力。

技术难点
问题表现
解决方法
多端协同复杂
项目同时包含用户端、管理端和服务端。
采用前后台分目录组织，前台走 `/api/public`、`/api/user`，后台走 `/api/admin`，接口职责清晰。
商品购买与服务预约链路差异大
商品走购物车和订单，服务走宠物+门店+时段预约。
前台拆分 `cart`、`booking` 两套 Pinia store，避免状态耦合。
数据一致性
下单涉及库存扣减，取消订单涉及库存回滚；预约涉及时段余量。
后端在 `orderService.js`、`bookingService.js` 中使用事务和快照字段保证一致性。
前后端数据模型不完全一致
后端字段为 snake_case，前端展示更偏视图模型。
使用 `src/adapters/` 统一做字段适配，降低页面复杂度。
移动端页面复用与品牌统一
页面数量多，容易风格碎片化。
通过 `AppShell`、`BottomTabBar`、`ProductCard`、`ServiceCard` 等组件复用统一 UI。
后台内容维护与图片上传
管理端既要做 CRUD，也要处理图片上传。
通过 `multer` 中间件解析上传，再由 admin API 返回图片 URL。

六、项目总结与反思
通过本项目可以更加直观地理解移动项目开发并不只是把页面做出来，而是要同时考虑业务链路完整性、状态管理边界、接口设计、数据持久化和后续维护成本。移动端页面数量一多，如果没有统一的组件与状态抽象，后期修改的代价会迅速上升。
本项目给出的经验主要有三点：第一，前期的页面结构设计和路由规划要足够清晰，否则商品链路和服务链路容易互相干扰；第二，前后端联调时必须有稳定的数据模型和适配层，否则页面实现会被后端字段变化牵着走；第三，测试不能只覆盖“能跑通”的理想路径，还要关注断言是否真正跟随业务演进同步更新。
从教训上看，项目虽然已经完成了较完整的前后台结构，但测试用例中仍有少量历史断言没有及时更新，说明“功能实现完成”并不等于“工程交付完成”。后续如果继续迭代，应该优先补齐测试、统一字段命名规范，并结合真实运行截图完善最终展示材料。
