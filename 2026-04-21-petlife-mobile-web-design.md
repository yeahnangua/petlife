# PetLife 宠物生活馆移动端 Web 作品集项目设计

日期：2026-04-21

## 1. 背景与目标

本项目要实现一个面向年轻养宠人群的移动端 Web 作品集项目，主题为 `PetLife 宠物生活馆`。项目核心不是“普通微商城换宠物商品”，而是把 `商品购买`、`服务预约`、`宠物档案` 和 `会员权益` 组合成一个更接近真实商业产品的体验。

项目分两阶段推进：

- 第一阶段由 Claude 输出前端页面与视觉界面，重点完成页面结构、品牌视觉、组件样式和移动端展示效果。
- 第二阶段由 Codex 补齐剩余逻辑，包括页面组织、状态管理、交互规则、表单校验、假数据组织、下单与预约流程串联。

第一版目标：

- 提供完整的移动端品牌化首页、分类、商品详情、服务详情、下单、预约、订单中心和个人中心页面。
- 明确区分两条主业务链路：`商品购买链路` 与 `服务预约链路`。
- 用 `宠物档案` 和 `会员权益` 两个模块拉开与普通商城项目的差异。
- 保持作品集导向的视觉质量，风格成熟、温暖、专业，避免过度卡通化。
- 所有核心页面都能够用静态数据跑通展示，便于先完成前端界面。

成功标准：

- 首页第一眼能够传达“宠物生活方式品牌”而不是普通电商模板。
- 用户可以清楚理解两条主流程：购买商品、预约服务。
- `宠物档案` 能够作为多个页面的辅助上下文出现，而不是孤立资料页。
- 页面之间的品牌语言统一，包括配色、圆角、卡片层级、图标风格和按钮样式。
- 后续补逻辑时，不需要大规模推翻 Claude 输出的页面结构。

## 2. 非目标

第一版明确不包含以下能力：

- 不接真实支付。
- 不接真实地图、定位、门店距离计算或导航。
- 不接真实后端和数据库；数据先采用本地 mock。
- 不做多商户入驻、多门店库存联动和复杂履约。
- 不做社区发帖、评论、私信等社交模块。
- 不做直播、拼团、分销、砍价等高复杂营销玩法。
- 不做复杂会员等级规则，只保留作品集足够展示的轻量权益。
- 不做后台管理系统；本项目只聚焦移动端用户侧。

## 3. 技术选型与协作边界

推荐技术栈如下：

- 前端框架：`Vue 3`
- 构建工具：`Vite`
- 路由：`Vue Router`
- 状态管理：`Pinia`
- 样式方案：`CSS Modules` 或 `Scoped CSS`，配合全局 design tokens
- 数据阶段：第一版以本地 mock 数据为主

选择理由：

- 该技术栈足够轻，适合移动端 Web 作品集项目。
- 页面与组件拆分清楚，便于先做视觉、后接逻辑。
- 如果 Claude 先输出 Vue 页面结构，后续 Codex 可以直接在相同技术栈中补逻辑，不需要大改。

协作边界：

- Claude 负责：
  - 页面结构
  - 视觉风格
  - 组件层级
  - 静态页面排版
  - 基于 mock 数据的展示型界面
- Codex 后续负责：
  - 路由组织
  - 页面串联
  - 状态管理
  - 购物车、预约单、订单流转逻辑
  - 表单交互与校验
  - mock service / API 抽象
  - 公共数据模型与页面行为统一

关键约束：

- 服务预约与商品购买在第一版中保持两条独立链路。
- `服务订单不进入购物车`，直接从服务详情页走预约提交流程。
- 购物车只承载商品，不混合服务，避免第一版范围失控。

## 4. 产品定位

### 4.1 品牌定位

`PetLife 宠物生活馆` 是一个面向城市年轻养宠用户的宠物生活方式型品牌，提供宠物食品、用品购买，以及洗护、美容、寄养等轻服务预约，并通过宠物档案与会员权益增强用户留存感。

项目调性：

- 温暖
- 专业
- 轻疗愈
- 生活方式感

不采用的方向：

- 不做儿童化卡通宠物风。
- 不做高饱和“萌系电商”风。
- 不做极简科技感黑白风。

### 4.2 目标用户

核心用户画像：

- 20-35 岁城市养宠人群
- 以猫犬用户为主
- 有稳定消费习惯
- 对品牌感、便利性和宠物健康有一定关注

### 4.3 核心使用场景

核心场景分为三类：

1. 日常购买宠物食品、零食、玩具、清洁用品
2. 预约洗护、美容、寄养等到店或托管服务
3. 记录宠物基础资料，并查看与宠物相关的推荐商品和服务

## 5. 信息架构与页面范围

### 5.1 一级导航

底部主导航建议为五项：

1. `首页`
2. `分类`
3. `服务`
4. `订单`
5. `我的`

原因：

- `服务` 单独成为一级入口，能直接体现项目差异化。
- `订单` 单独成 tab 有利于展示商品订单与服务订单双类型。
- `宠物档案` 收纳在“我的”内，作为用户资产模块，而不是单独 tab。

### 5.2 核心页面清单

第一版完整范围建议控制在 `10-13 个页面`，推荐优先级如下：

1. 首页
2. 分类页
3. 商品列表页
4. 商品详情页
5. 服务首页 / 服务列表页
6. 服务详情页
7. 购物车页
8. 商品确认订单页
9. 服务预约确认页
10. 订单中心页
11. 我的页面
12. 宠物档案页
13. 会员权益页

范围说明：

- 实现时可将 `商品列表页` 视为分类页的二级状态，以减少页面数量。
- `会员权益页` 可先作为“我的”页内的二级详情页处理。
- 如果时间紧，第一版最低可保留 10 个页面：把会员权益页和宠物档案页做成“我的”里的分区入口。

## 6. 核心业务链路

### 6.1 商品购买链路

链路如下：

`首页 / 分类页 -> 商品列表 -> 商品详情 -> 加入购物车 -> 购物车 -> 确认订单 -> 支付成功态展示 -> 商品订单详情`

第一版规则：

- 商品支持多规格选择，例如重量、口味或适用品类。
- 商品详情页可加入购物车，也可直接购买。
- 确认订单页支持地址、优惠券、备注、配送方式的静态展示与基础交互。
- 支付只做结果态模拟，不接真实支付。

### 6.2 服务预约链路

链路如下：

`首页 / 服务页 -> 服务详情 -> 选择宠物档案 -> 选择门店 / 日期 / 时间段 -> 填写备注 -> 提交预约 -> 服务订单详情`

第一版规则：

- 服务不进入购物车。
- 预约需要依赖已存在的宠物档案；若无档案，流程中引导添加。
- 时间段与门店均使用 mock 数据。
- 提交后进入“待服务”订单状态。

### 6.3 宠物档案辅助链路

辅助链路如下：

`我的 -> 宠物档案 -> 查看 / 新增 / 编辑宠物资料 -> 返回服务预约或推荐模块`

第一版中，宠物档案会影响：

- 服务预约时的宠物选择
- 首页的猫犬推荐内容切换
- 商品详情中的适用建议文案

## 7. 页面设计说明

### 7.1 首页

首页承担品牌展示、业务分流和推荐承接三类职责。

推荐模块：

- 品牌头部：Logo / 品牌名 / 搜索框 / 消息或会员入口
- 宠物切换：猫 / 狗双态切换
- 首屏 Hero Banner：品牌视觉图 + 主 CTA
- 快捷入口：主粮、零食、玩具、洗护、美容、寄养、会员、新人礼包
- 热门服务区：大卡片形式突出预约能力
- 场景组合推荐：新手养猫、换季护理、出行套装等
- 热卖商品区：标准商品卡片列表
- 会员权益卡：积分、优惠券、新人礼包摘要

首页目标：

- 先传达品牌感，再引导业务，不做“满屏促销图”。
- 保持留白，控制信息密度，避免模板化商城观感。

### 7.2 分类页

分类页以商品消费为主，结构要简单直接。

建议结构：

- 左侧一级分类：猫 / 狗 / 通用
- 右侧二级分类和商品卡片列表
- 顶部保留搜索与筛选入口

分类建议：

- 主粮
- 零食
- 玩具
- 清洁
- 出行
- 保健

### 7.3 商品详情页

商品详情页是商品链路的核心展示页。

建议模块：

- 商品轮播图
- 标题、副标题、价格、会员价
- 规格选择
- 适用宠物标签
- 功效 / 卖点摘要
- 图文详情
- 用户评价摘要
- 搭配推荐
- 底部悬浮操作栏：加入购物车、立即购买

差异化设计：

- 基于宠物类型显示“适合 1-3 岁成猫”等建议。
- 增加“与最近预约服务更搭配”的关联推荐文案。

### 7.4 服务页

服务页应明显区别于商品列表页，不应只是“换了字段的商品卡片”。

建议模块：

- 服务专题 Banner
- 常用服务入口
- 热门服务卡片
- 服务说明小贴士
- 门店或服务保障信息

服务类型示例：

- 洗护清洁
- 精致美容
- 驱虫护理
- 短期寄养

### 7.5 服务详情页

服务详情页承担服务说明和预约转化。

建议模块：

- 服务主视觉
- 服务名称、时长、价格、评分
- 服务包含内容
- 适用宠物说明
- 注意事项
- 可预约门店信息
- 可选时间段展示
- 底部主 CTA：立即预约

### 7.6 购物车页

购物车只承载商品，保持结构清晰。

建议模块：

- 商品分组列表
- 规格摘要
- 数量加减
- 失效商品状态
- 猜你喜欢
- 底部结算栏

### 7.7 商品确认订单页

建议模块：

- 收货地址卡片
- 商品列表摘要
- 优惠券选择
- 配送方式
- 订单备注
- 金额明细
- 提交订单按钮

### 7.8 服务预约确认页

建议模块：

- 已选服务摘要
- 宠物选择卡片
- 门店选择
- 日期与时间段
- 预约说明
- 联系电话
- 备注
- 提交预约按钮

关键交互：

- 无宠物档案时，展示“先添加宠物档案”的引导态。

### 7.9 订单中心页

订单中心需要明确区分 `商品订单` 与 `服务订单`。

建议结构：

- 顶部切换：全部 / 待支付 / 待发货 / 待收货 / 待服务 / 已完成
- 商品订单卡片
- 服务订单卡片
- 空状态插画与引导

### 7.10 我的页面

建议模块：

- 用户信息头部
- 会员积分摘要
- 优惠券入口
- 宠物档案入口
- 我的订单快捷入口
- 常用功能入口：地址、收藏、设置

### 7.11 宠物档案页

宠物档案是本项目的代表性特色模块之一。

字段建议：

- 宠物头像
- 昵称
- 品种
- 性别
- 年龄
- 体重
- 生日
- 过敏信息
- 常用服务偏好

页面形态建议：

- 宠物档案列表
- 单个档案卡片详情
- 新增 / 编辑表单

### 7.12 会员权益页

会员页不做复杂等级系统，只保留轻量展示型权益。

建议模块：

- 当前积分
- 可用优惠券
- 新人礼包
- 会员专享价格说明
- 权益说明卡片

## 8. 视觉设计方向

### 8.1 视觉关键词

项目视觉关键词如下：

- 温暖
- 专业
- 轻疗愈
- 有品牌感

目标观感：

- 像一个真实存在的宠物生活方式品牌
- 像高质量作品集项目
- 不像通用商城模板

### 8.2 颜色系统

建议颜色方向：

- 主色：`鼠尾草绿`
- 辅助底色：`奶油白`、`燕麦色`
- 强调色：`珊瑚橙` 或 `琥珀黄`
- 文字主色：深灰
- 弱化文字：灰棕色系

建议避免：

- 高饱和粉蓝宠物店配色
- 纯黑白极简科技风
- 纯紫色系互联网模板配色

### 8.3 组件与版式风格

组件风格建议：

- 大圆角卡片
- 柔和阴影
- 大图与留白
- 品牌插画或高质量宠物照片点缀
- 强 CTA 使用高对比但不刺眼的强调色

版式建议：

- 移动端优先设计宽度：`375px` 与 `393px`
- 内容区以卡片流和分区模块为主
- 首屏避免堆叠过多运营元素
- TabBar 与底部操作栏要有足够安全区

### 8.4 图标与图片风格

建议：

- 图标采用线性圆角风格
- 插图仅用于空状态与轻运营区
- 商品与服务主图尽量采用高质量实拍或高完成度 mock 图

## 11. 交互与状态规则

第一版应明确以下关键状态：

- 加载态
- 空状态
- 失效状态
- 表单未完成态
- 提交成功态

重点规则：

- 购物车为空时，提供商品推荐入口。
- 无宠物档案时，服务预约页必须引导新增档案。
- 商品详情规格未选完整时，不允许直接购买。
- 预约日期或时间段未选时，不允许提交预约。
- 订单中心按订单类型和状态过滤时，卡片样式保持一致，只切换信息层级。

## 12. 错误处理与兜底策略

由于第一版以 mock 数据为主，错误处理重点放在 UI 和交互兜底，而不是网络异常恢复。

第一版应覆盖：

- 数据为空时的空状态页
- 图片加载失败时的默认占位
- 表单缺失时的前端校验提示
- 商品失效时的卡片置灰和文案提示
- 服务时间段无库存时的禁用态展示

## 13. 测试与验收重点

第一版验收重点如下：

- 页面在主流移动端宽度下不出现明显错位
- 首页、商品详情、服务详情三个关键页面视觉统一
- 商品链路能完整跑通到下单结果态
- 服务预约链路能完整跑通到预约结果态
- 宠物档案可被服务预约流程引用
- “我的”页能串联会员、档案、订单入口

## 14. Claude 前端输出约束

将本设计交给 Claude 输出页面时，应明确以下要求：

- 以 `Vue 3` 单文件组件或等价的 Vue 页面结构输出
- 先完成页面静态结构与样式，不接后端
- 使用统一的 design tokens 管理颜色、圆角、阴影和间距
- 所有页面移动端优先，不先做桌面端
- 组件尽量复用，避免每个页面各写一套卡片
- 保留 mock 数据入口，便于后续由 Codex 替换为状态管理和逻辑层

不建议对 Claude 的要求：

- 不要求它在第一轮就完成复杂业务逻辑
- 不要求它接真实接口
- 不要求它处理复杂状态同步

## 15. 分阶段实施建议

建议按以下顺序落地：

1. Claude 先完成品牌视觉与核心页面静态稿
2. 优先输出首页、商品详情、服务详情、我的页面四个关键界面
3. 再补分类、购物车、确认订单、预约确认、订单中心
4. Codex 接手整理目录、路由、mock 数据结构和状态管理
5. Codex 补齐商品购买链路与服务预约链路
6. 最后统一交互细节、空状态和表单校验

## 16. 设计结论

`PetLife 宠物生活馆` 的第一版应该被定义为一个 `商品购买 + 服务预约 + 宠物档案 + 会员权益` 的移动端作品集项目。

它的关键不在于功能堆叠，而在于：

- 业务结构足够真实
- 页面类型足够丰富
- 视觉调性有品牌辨识度
- 与普通微商城相比，能明显体现服务型产品思维

为了控制范围，第一版明确采用以下策略：

- 商品和服务分两条链路
- 服务不进购物车
- 用 mock 数据先完成高质量前端展示
- 后续再补齐状态与逻辑层
